Project details:
- We are writing a C program that "hex dumps" a given file.
- The given file is output in the following format:
  - The number of bytes in the file, in hex (i.e. the memory address)
  - 16 consecutive bytes, in hex, separated by spaces
  - ASCII representation of the bytes, with . replacing non-ASCII bytes

    Fig. 1: Example output of the program

    00002040 75 6e 74 75 20 31 33 2e 33 2e 30 2d 36 75 62 75 |untu 13.3.0-6ubu|
    00002050 6e 74 75 32 7e 32 34 2e 30 34 2e 31 29 20 31 33 |ntu2~24.04.1) 13|

- Additionally, we are compiling said program with optimization options -O0, -O1 and -O3.
- We will analyze and discuss the different ASM code generated by different optimization levels.

Analysis of ASM from -O0 compilation:
- O0 is no optmiization, and should just "translate" our C into ASM.
- Despite its relative lack of optimization, the compiler still knows to turn our string constants into variables. | is saved as a constant.
- O0 moves useless variables to and from the registers when it doesn't need to. For example, it moves "i" multiple times, when in reality it's just used for a loop.

Analysis of ASM from -O1 compilation:
- O1 is balanced optimization.
- O1 uses tricks with testq for earlier validation checks against NULL.
- O1 doesn't bother with moving useless variables too often, and uses cmpb functions to test instead of mov, which is generally more effiient.

Analysis of ASM from -O3 compilation:
- O3 aggressively optimizes for speed, sacrificing size optimizations. We can see that proj3_O3.s is 2x the size of its priors!
- O3 doesn't use as many jmp instructions as O1 and O2, instead just writing the hex dumps themselves 16 times. This is presumably faster.
